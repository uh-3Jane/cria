@realShaman | DefiLlama regarding this the problem is actually the following: 

The current stacks-api.js uses fetchCallReadOnlyFunction from @stacks/transactions, which doesn't expose the ?tip= param that the underlying /v2/contracts/call-read/ endpoint supports. The rest of the Clarity serialization can stay as-is. Happy to submit a PR for this change to stacks-api.js if that'd help, it's a small change and would unblock timetravel for all stacks adapters.

```text
yes, it’s actually to prevent double counting. In reality, Ethena has wallets that redeposit collateral into Aave, which significantly inflates the TVL.

Imagine a user mints USDe using their USDC. Then the user deposits their freshly minted USDe into Aave.

On Ethena’s side, they now hold the USDC. Part of it is sent to CEXs for basis trading or other yield-generating strategies. Another portion is kept as reserves and redeposited into Aave.

So you end up with the user depositing USDe into Aave, and Ethena’s wallets depositing the USDC backing those USDe into Aave as well. This means that for an initial 1 USDC from a user, you can end up with more
```

- it does not run `/scan`

- chatbot is meant to be a lightweight support concierge
- scan/issue tracking remains the real source of truth
